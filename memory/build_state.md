**PROJECT: Career Player Comp.** A web app where a user submits their career history (LinkedIn PDF upload OR plain-text paste), answers 8 oblique career questions + 2 optional text inputs, and Claude maps them to ONE real NBA player from a curated pool, returning a personalized "scouting report": player name, a freshly-coined archetype title, 2K-style badge pills, a fake stat line, and 2-3 sentences of sharp copy that MUST cite specifics from their real career. Output is a shareable card (web + Instagram Stories formats). Every output is generated fresh by Claude; no pre-written player paragraphs. Explicit design goal: blue-collar/trade careers treated with the SAME analytical respect and specificity as executive ones (nurse, electrician, forklift operator, chef, vet, warehouse lead, CEO all get real, specific comps). FRAMING (Maurice's correction): equal, specific treatment of every career is BASELINE decency, NOT the product's unique hook — do not oversell it or center the marketing on it. The real hooks are accuracy, personalization, ease of use, and the novelty of the concept. The only hard rule here: never hand a non-office/trade career a dismissive or generic comp; treat all careers equally and specifically.

**This is Maurice's SECOND project.** The first, Pros Are Fans, is built through Backend Increment A and paused-not-abandoned at `/Users/maurice/Desktop/AI development/pros-are-fans` (fully saved/resumable). We chose to do Career Player Comp FIRST because it's a fast (~2-3 week), public, shareable visibility win during his active job-search/LinkedIn-rebuilding window, where Pros Are Fans is stealth and slow-burn. Treat this as a time-boxed sprint that SHIPS; the risk to manage is project-hopping, so finishing matters.

**LOCKED DECISIONS (Maurice approved):**
1. **Model = Claude Sonnet 4.6** (`claude-sonnet-4-6`), NOT Haiku. The whole product is comp quality (right player + specific, earned copy); Haiku risks generic output. Use prompt caching on the system prompt + player JSON (90% discount). Test Haiku side-by-side on ~10 real profiles later and only downgrade if it genuinely holds the bar. (Use the `claude-api` skill when building the API integration — it handles caching + current model best practices.)
2. **Store nothing** for launch (no inputs, no outputs). Clean privacy promise. Revisit an anonymized-outputs gallery later.
3. **Tone dial = witty, snarky, can sting a bit, but ACCURACY ABOVE ALL.** The real-career-to-real-player-history parallels are the point and must feel undeniably accurate. The sting should read as "damn, that's true and I respect it," never as an insult that kills sharing. Blue-collar dignity framing throughout; no comp framed as failure. Test on real humans for emotional response, not just accuracy.
4. **Both native share (Web Share API) AND PNG download** on results.
5. **Donation copy reframed:** not "Buy a Credit ($2)" (implies a purchased credit). Frame as an honest tip, e.g. "Chip in $2 to keep it running." Stripe Payment Link (external URL), no SDK.

**ADJUSTMENTS to the pasted build spec (apply these; they override the spec where they conflict):**
- **Card export: build it server-side** (satori / Next-OG-style image generation), NOT html2canvas. html2canvas mis-renders custom fonts/shadows/CSS texture, and the card is the entire growth engine. Server-side gen is pixel-perfect and yields the share-preview/OG image for free. (Note: spec assumes Vite SPA; reconcile during planning — either add a small server image route, or reconsider Next.js if it makes the card-gen + edge proxy cleaner. Decide at plan time.)
- **Rate limiting needs real plumbing:** stateless edge functions can't IP-limit alone. Use a tiny KV store (Upstash free tier) for the per-IP limit (≈5/IP/hr), AND add a separate GLOBAL daily spend cap as a hard kill-switch so a viral spike can't run up a bill.
- **Player POOL is the real moat,** and DEPTH + COVERAGE matter more than fame (Maurice's instinct, validated with refinements): (a) go deeper than 150 — target ~200–250 (sweet spot; past ~250 you hit diminishing returns and matching DILUTION as the model weighs too many options); (b) include MANY MORE DEEP CUTS and ALL ERAS (1960s–present) — an obscure-but-perfect comp delights ball-knowers and is more on-brand than another superstar; (c) the real goal is covering the full space of CAREER ARCHETYPES (one-team lifer, journeyman survivor, late bloomer, undrafted overachiever, pure specialist, injury-derailed, system-dependent, locker-room glue, enforcer, energy-bench spark, international import, positionless utility, comeback, trade-deadline rental, coach-on-the-floor, etc.) so EVERY user maps cleanly — count serves coverage, not the reverse; (d) each player (ESPECIALLY deep cuts) needs a RICH, ACCURATE `work_style`/`career_traits` — the model matches on the characterization text, not the name, and both the model and the curation can be shaky or hallucinate on obscure players, so accuracy-check the deep cuts; (e) STRONGLY CONSIDER including WNBA legends (Taurasi, Sue Bird, Maya Moore, Lisa Leslie, Swoopes, Catchings, Fowles, Candace Parker, Becky Hammon) — the brand name isn't NBA-specific, so it fits cleanly, broadens appeal, avoids comparing women only to men, and matches the dignity ethos. Default: include them unless Maurice vetoes. Thin tags = mush matches regardless of pool size.
- **Quiz needs back-navigation** (auto-advance on tap is fine, but allow changing a mis-tapped answer).
- **Input = clean career data, lowest friction, NO whole-page dumps.** Do NOT claim "paste is easier" (it isn't universally; that was a wrong earlier framing). Offer two equal paths, both framed to elicit CLEAN input: (1) UPLOAD a résumé OR a LinkedIn "Save to PDF" export — a résumé is often the cleanest, most on-point career doc, so accept it first-class, not just LinkedIn; (2) PASTE work history or résumé text, with a placeholder showing the clean format (Title, Company, Years) and an explicit guard NOT to paste the whole LinkedIn page (modern profiles are cluttered with featured posts, recommendations, "people also viewed," and activity — pure noise that confuses the model). If a paste is huge or looks like a full page, gently warn to trim to work history. Extraction/parse < 50 chars → surface the paste path. The system prompt must extract Experience + Education (+ the user's paragraph) and IGNORE résumé/LinkedIn cruft.
- **Output validation + one retry** if Claude returns a player not in the pool or invalid JSON. Keep all the no-AI-tells / no-em-dash / no-purple / no-gradient rules from the spec (they already match Maurice's standards).

**ACCURACY, CONNECTIVE TISSUE & WNBA:**
- WNBA legends are in the pool and assignable to ANY user regardless of presumed gender. The system prompt must NOT gender-gate the comp.
- **Player-trait accuracy:** Maurice personally QAs the player characterizations — he ran NBA coverage at ESPN and is a genuine domain expert, so his eye is the accuracy backstop, especially for deep cuts. Process: model drafts the pool, then a separate verification pass plus Maurice's review checks each player's real career arc/role and flags uncertain ones; fix or drop. Never ship a characterization that neither the model nor Maurice is confident is accurate.
- **Make the connective tissue EXPLICIT:** the scouting_report must name the specific career-to-player parallel (the actual reason for this player), not just assert a vibe, so the user can judge the logic. If the "why" isn't legible and true, the comp feels random. This both raises perceived accuracy and surfaces bad matches during the spike.

**UX & VIRAL-TOOL PITFALLS (mobile-first; the product wins on accuracy + personalization + ease + uniqueness):**
- Front-load the payoff: player name + coined title land in the first ~2 seconds on the result; the card is the punchline.
- Upload is the #1 drop-off point: paste is first-class, the LinkedIn how-to leads with the fast method, and trust is reassured at the PDF step (nothing stored; data is sent to the AI only to generate the comp).
- Sharing must be frictionless: native share + the server-rendered card-as-image (no html2canvas surprises); the shared link's OG image must be compelling (free from server-gen).
- Keep the call fast (Sonnet, max_tokens 800, prompt caching) so the "scouting room" theater covers a short wait, not a long one; handle API/cost spikes gracefully (KV rate limit + global spend kill-switch + a "front office is backed up" backoff).
- Same input yields the same comp (low temperature): a reroll that changes the player kills the accuracy trust.
- Test for the mean/uncanny failure: a comp that stings the wrong way doesn't get shared; judge emotional response, not just accuracy.
- Mobile-first at 375px: touch targets ≥44px, no horizontal scroll on the result card, both share/download buttons fit, respect prefers-reduced-motion, check dark-theme contrast.

Full screen flow + the 8 questions + system prompt + player-pool distribution are in the build spec Maurice will paste into the build session.

**OUTPUT QUALITY IS THE #1 RISK — validate it BEFORE building the app.** The product is worthless if the comps aren't apt, specific, and witty; the share card only matters if the comparison lands. LLMs gravitate to generic/famous comps and to fake-specificity, so do NOT assume quality; prove it cheaply first (it's just API calls, a few cents). Test EDGE profiles too, not just clean ones: a 3-industry career-changer, a 25-year one-company lifer, a 2-year early-career person, and a messy gig/freelance patchwork. Use at least 2 REAL profiles (Maurice's own LinkedIn + a real tradesperson if possible); synthetic profiles flatter the model and hide failure modes. DETERMINISM: keep temperature low so the same input yields a consistent, authoritative comp — a reroll that changes the player undercuts the "accuracy" feel; frame "Appeal Your Report" as "give the scout more to go on" (add the paragraph), not a random reroll.
